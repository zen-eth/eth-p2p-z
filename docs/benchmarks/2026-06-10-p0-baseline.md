# P0 baseline — gossipsub router + QUIC loopback micro-benchmarks

**Date:** 2026-06-10 · **Tree:** post-P1 (`c0dc23d` + benches)
**How to run:** `zig build bench-gossipsub -Doptimize=ReleaseFast`,
`zig build bench-quic -Doptimize=ReleaseFast`

These are the measurement baselines the perf backlog (review Part III/V) is
ranked against. Two environments matter: macOS/kqueue (dev) and Linux/epoll
(production shape). Numbers below: **macOS, Apple Silicon (10 executors)**;
Linux numbers from the same benches should be appended when run.

## Gossipsub router (in-process, discard transport, mesh=8, 256 B payloads)

| scenario | msgs/s (validated) | µs/msg | delivered/unique | allocs/offered |
|---|---|---|---|---|
| strict / inline (`concurrency=0`) | 19.6 k | 50.9 | 20000/20000 | 13.0 |
| **strict / async paced** (`=executors`) | **58.9 k** | **17.0** | 10376/20000¹ | 11.4 |
| strict / async **unpaced burst** | 14.9 k | 67.0 | 601/20000¹ | 1.6 |
| strict / async dup×4 paced | 24.1 k (unique) | 41.5 | 3520/5000¹ | 8.1 |
| anonymous / inline (no crypto) | 57.9 k | 17.3 | 20000/20000 | 9.0 |

¹ shortfall = go-parity validation-throttle drops; msgs/s is computed from
DELIVERED messages over wall time, so it is the sustained validated rate.

**Readings:**

- **P1 verified by measurement: async verification = 3.0× inline (58.9 k vs
  19.6 k) and matches the no-crypto ceiling (57.9 k)** — with the worker
  pipeline, Ed25519 costs the router fiber nothing; the router fiber itself
  (decode + dedup + fan-out to 7 sinks + delivery) is now the limit at ~17 µs/msg.
- **New finding (burst coupling):** under an unpaced burst, the async pipeline
  collapses (601/20000 delivered): verdict re-entry commands queue in the SAME
  FIFO inbox behind the flood of ingress, so in-flight slots are not freed and
  ~97 % of the burst throttle-drops. go does not have this coupling (its workers
  complete without the processLoop's help). This is direct evidence for the
  review's two-inbox recommendation (Part IV (d)): a small control/verdict queue
  drained with priority over data ingress. Until then: real network ingress is
  peer-paced and far below these rates, but a deliberate flood degrades async
  throughput below inline.
- allocs/offered ≈ 13 (strict) / 9 (anonymous) confirms the review's ~10
  allocations-per-message estimate (P7).

### Heartbeat (seen = 100 000 ids)

| tick | duration |
|---|---|
| steady (full-map scan, nothing expires) | 0.7–1.3 ms |
| **expiry tick (all 100 k age out at once)** | **3 247 ms** |
| post-expiry | 0.01 ms |

**Reading:** the P4 `O(n·k)` restart-after-each-removal sweep is measured at
**3.2 SECONDS** of router-fiber stall for a 100 k-id expiry. At eth2 steady
state (~180 k retained ids, ~1.5 k expiring per tick) the per-heartbeat cost is
the same order — the expiry-wheel replacement (review P4) is not optional at
mainnet rates.

## QUIC loopback (two real endpoints, full TLS handshake, single stream)

| scenario | goodput | rx pps (server) | drops |
|---|---|---|---|
| bulk, 64 KiB writes (256 MiB) | 0.130 Gbit/s | 21.5 k | 0 |
| small, 1200 B writes (32 MiB) | 0.116 Gbit/s | 23.7 k | 0 |

**Reading:** ~21–24 k packets/s regardless of write size = the per-packet
fixed-cost ceiling the review identified (one event-loop round trip per packet
on send — zio has no sendmmsg/GSO — plus 2 fiber spawns + 1 cancel + 1 heap
dupe per packet on recv). Production QUIC stacks do multi-Gbit/s on loopback;
the ~20×+ gap is the P2 (send batching/GSO) + P3 (persistent recv fiber + slab)
budget. Re-measure after each to attribute the gain.

|---|---|---|---|
| bulk 64 KiB | 0.091 Gbit/s | 12.7 k | 0 |

**Cross-platform readings:** every finding reproduces — P1 async = 3.0–3.3×
inline on both; the anonymous (no-crypto) router-fiber ceiling is ~57 k msgs/s
on BOTH machines (the fan-out path, not the CPU, bounds it); the burst-coupling
collapse and the multi-second expiry stall reproduce; the QUIC per-packet
ceiling is even lower on the shared Codespace (12–13 k pps), reinforcing that
per-packet fixed cost, not bandwidth, is the limiter.

## After P3 (persistent recv fiber + shared CID table + slab pool)

Same benches re-run on the P3 tree (`67cf084` + `3ee2bb1`):

| QUIC scenario | macOS (10c) | Linux/epoll (16c) | vs baseline |
|---|---|---|---|
| bulk 64 KiB | 0.130 → **0.892 Gbit/s** | 0.091 → **0.729 Gbit/s** | **6.9× / 8.0×** |
| small 1200 B | 0.116 → **0.913** | 0.062 → **0.689** | **7.9× / 11.1×** |
| server rx pps | 21.5 k → **88.5 k** | 12.7 k → **72.3 k** | 4.1× / 5.7× |
| drops | 0 | 0 | — |

Stage attribution (macOS): stage 1 alone (persistent fiber + direct CID table,
no slab) delivered the full 7× — the per-datagram two-spawn+cancel Select was
the bottleneck itself, not merely overhead. Stage 2 (slab views) is throughput-
neutral on this loopback bench but removes the per-packet malloc/free and the
Linux GRO re-dupe pathology (~45 allocs+copies per 64 KB super-datagram → 0).
Gossipsub bench numbers are unchanged by P3 (different subsystem) — no
regression.

**Known bench bimodality (open investigation):** the macOS-loopback bulk
scenario is BIMODAL — the same binary measured 0.873 / 0.902 / **0.066** /
0.909 Gbit/s across four consecutive runs. The slow mode runs with wall times
quantized at ~4.25 ms per 64 KiB chunk (a delayed-ACK/pacing lockstep shape)
and is entered by initial timing, not by code version: it was first
misattributed to multi-datagram slab sizes during P3 stage 2 (every
configuration, including slab-off, can land in either mode). Treat single bulk
runs as 3-of-4 reliable; rerun outliers. Root-causing the lockstep (pacing vs
delayed-ACK vs timer granularity — one keylog+tshark capture of a slow run
should pin it) is an open backlog item. The one-datagram slab default stands
on its own merits (pinning granularity); larger bump-arena slabs remain
available via `recv_slab_slot_bytes` and are no longer suspected.

## After the two-inbox split (control/verdicts prioritized over data ingress)

The burst-coupling finding above is fixed: validation verdicts, peer
lifecycle, heartbeat, and shutdown now ride a CONTROL inbox drained completely
before each data command (wakeups via a best-effort `.control_ready` marker in
the data queue — full data queue ⇔ consumer already awake, so no extra
primitive and no lost wakes). macOS re-measurement:

| scenario | before | after |
|---|---|---|
| strict/async **burst** delivered | 601/20000 (self-starved) | **1977/20000**, 14.9 k → **33.4 k msg/s** |
| strict/async paced delivered | 10376/20000 | **15502/20000** |
| strict/async dup×4 paced | 3520/5000 | **4756/5000 (95 %)** |
| inline / anonymous | — | unchanged (no regression) |

Remaining burst shortfall is the REAL capacity gap (offered at memcpy speed vs
verification capacity) — go-parity throttle semantics, not starvation.

## Per-stage counters still missing (follow-up)

The benches measure from the harness; the router itself still has no
first-class counters for validation-throttle drops, seen-cache size, or
per-peer lane drops. Worth adding alongside the P2/P3 work so production nodes
can see what the bench sees.
